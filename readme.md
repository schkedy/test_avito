Проблемы с которыми я столкнулся и их решения:

1) В API спецификации отсутствовали критически важные Responses, решено было их добавить и обработать (400,415,500)
2) Так как в изначальной спецификации было написано что /team/add добавляет и ОБНОВЛЯЕТ команды, то решил убрать ошибку Team Exists и обновлять команду в соответсвии с переданными данными, принципом замены. Если команда создается то возврщала 201 если обновляется то 200. Создание/обновление команды переназначет существущих юзеров в  новую команду.
3) В example на reasign былл old_revier_id вместо old_user_id, заменил на требуемое название (2).
4) Исходя из дополнительного задания предпологаю что user могут быть назначены только к одной команде. 
5) Решил добавить /pullRequest/assign чтобы назначать случайных/указнных ревьюеров только на pr без ревьюеров.
6) Решил не делать транзакцию в бд на сбор статистики, хотя сбор статистики и читает из разных таблиц. Слышал что сбор метрик не должен влиять на ход работы, а транзакция достаточно дорогостоящая процедура. Особенно если в будущем этот эндпоинт будет использоваться для более обширного сбора метрик через prometheus, допустим.
7) Решил что слоистая архитектура лучше всего подходит, так как удобно поддерживать, разрабатывать и изменять
8) Добавление и обновление команды выполняется транзакцией, чтобы избежать нелепых ситуаций. Добавление пул реквеста тоже транзакция, так как структура бд подразумевает хранить еще отноешения pr и user в отдельной таблице, и автоматически назначать. Переназначение, назначение ревьюера тоже транзакция
9) Sqlc так как избавляет от огромной рутины, oapi-codegen по той же логике, gin удобные middleware для логирования,отлаваливания паники, медленее чем chi chi/ nethttp, но под требование для средних данных хватит с лихвой. Viper - гибкая настройка окружения, pgx vs lib/pq пока еще не разобрался что на самом деле лучше, решил выбрать pgx потому что звучит круче. Slog для Sтруктурных и сквозных логов.
10) Добавил в спецификацию локальный url сервера, чтобы можно было вручную тестировать через Insomnia или любой другой клиент 
11) Нагрузочное тестирование првоодил через k6, позволяет писать сложные сценарии.
12) добавил еще два эндпоинта /stats и /health
13) Описание бд в docs/schema.pdf (КАРТИНКА!) с помощью https://dbdiagram.io
